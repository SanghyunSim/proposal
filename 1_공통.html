공통 시스템 분석 보고서

1. 목표 시스템 정의

1-Liner Definition

공통 영역은 KITA 차세대 무역플랫폼 전체 서비스가 동일한 품질·보안·운영 기준 위에서 안정적으로 확장되도록 클라우드 랜딩존, 공통 운영환경, 표준 개발체계, 테스트·전환·보안·품질·프로젝트 관리 거버넌스를 통합 제공하는 플랫폼 기반 시스템이다.

1.1 사업 배경 (Why)

  • 개별 서비스 중심 운영의 한계: Trade Digital Brain, Trade Agent, 통합대시보드, 수출실적증명, 지재권, tradeKorea, ESG, 무역아카데미가 동시에 구축되므로 서비스별 개별 인프라·보안·운영 기준으로 접근하면 장애 대응, 변경관리, 비용관리, 감사 추적이 분산된다. 공통 요구사항은 이를 통합 기준으로 묶기 위한 기반 요구다 [RFP III-가, p.17~29]
  • 클라우드 전환 리스크: RFP는 멀티 Account 기반 랜딩존, 사용자 관리, 네트워크, 고가용성, 확장성, 자원 증감, 비용 최적화까지 요구한다. 이는 단순 서버 이관이 아니라 클라우드 운영모델 자체를 설계하라는 의미다 [RFP III-가-ECR-CM-003, p.19]
  • 보안·감사 요구의 수평 적용 필요: 관련 법령·지침 준수, 시큐어코딩, 접근통제, 암호화, 서버·네트워크 보안, 로그·이력관리가 전 서비스에 공통 적용되어야 한다 [RFP III-가-SER-CM-001~005, p.23~24]
  • 품질·검수·감리 대응의 표준화 필요: 요구사항 변경관리, 형상관리, 검수지표, 감리 대응, 산출물 관리가 명시되어 있어 개발 산출물과 테스트 결과가 추적 가능한 체계로 관리되어야 한다 [RFP III-가-QUR-CM-001~004, p.25], [RFP III-가-COR-CM-005, p.27]

1.2 구축 목적 (What)

RFP I-4 추진범위의 공통 인프라 항목 [RFP I-4, p.3]:

"클라우드 랜딩존, 공통 운영환경, 보안체계 구축을 통해 인프라 고가용성 및 확장성 확보"

RFP III-가 공통 요구사항 요약 [RFP III-가, p.17]:

공통 영역은 시스템 장비구성 3개, 기능 1개, 성능 1개, 테스트 7개, 보안 5개, 품질 4개, 제약 6개, 프로젝트 관리 9개, 프로젝트 지원 4개 등 총 40개 요구사항으로 구성된다.

ECR-CM-003 클라우드 인프라 구축 [RFP III-가-ECR-CM-003, p.19]:

"멀티 Account 환경 기반의 랜딩존을 구축"하고, "인프라는 고가용성, 확장성 기반으로 구성"하며, "AI Agent 서비스에서 사용하는 MCP 서버"를 구축해야 한다.

1.3 최종 지향점 (Goal)

  • Cloud-Native Foundation: 멀티 Account 랜딩존, 네트워크, 보안, 관제, 백업·복구, 비용관리, Auto Scaling이 표준화된 전사 공통 클라우드 기반
  • Secure-by-Design Platform: 시큐어코딩, 접근통제, 암호화, 로그·감사, 인력 보안, 산출물 보안이 개발·테스트·운영 전 단계에 내재화된 체계
  • Quality-Gated Delivery: 단위·통합·성능·시스템·전환 테스트가 요구사항 추적성과 결합된 품질 게이트 방식으로 운영되는 구축 체계
  • Reusable Platform Core: 전자정부 프레임워크, 공통코드, 공통기능, 공통모듈, 표준 API, MCP 서버를 재사용 가능한 플랫폼 자산으로 축적

3. 핵심 요구사항 및 필수 고려사항 TOP 5

공통 영역은 개별 기능 배점보다 전체 사업의 수행 가능성·보안성·품질성·운영 안정성을 좌우한다. 평가표상 기술 및 기능 25점, 보안 요구사항, 데이터 요구사항, 성능 및 품질 15점, 프로젝트 관리 15점, 프로젝트 지원 20점과 직접 연결되므로 제안서에서 구조화된 구현 방안과 증적 중심 운영계획을 제시해야 한다 [RFP 제안서 평가기준, p.95~97].

KEY-1. 클라우드 랜딩존 + 공통 운영환경 구축

근거: [RFP III-가-ECR-CM-003, p.19]

  • 멀티 Account 환경 기반 랜딩존 구축
  • 사용자 관리 체계, 네트워크, 관리방안 포함
  • 고가용성·확장성 기반 인프라 구성
  • 자원 증감, 모니터링, 비용관리, 백업·복구, DMZ·WAF·IPS·UTM 구축
  • AI Agent 연계용 MCP 서버 구축

제안 배점 추정: 높음

  • 전 서비스의 실행 기반이므로 인프라 아키텍처, 운영모델, 보안모델, 비용관리 체계를 동시에 설명해야 함
  • 단순 CSP 리소스 나열이 아니라 Landing Zone Blueprint, Account Strategy, Network Segmentation, Security Baseline을 제시해야 평가 방어 가능

구현 핵심 포인트:

  • Shared, Security, Network, Logging, Workload-Dev, Workload-Prod 계정 분리
  • IaC 기반 표준 배포: Terraform + GitOps
  • 중앙 로그 계정 및 보안 계정 분리
  • DMZ, Private Subnet, NAT, WAF, IPS/UTM, Bastion 또는 접근통제 연계
  • 비용 태깅 표준: service, env, owner, cost-center, project

KEY-2. 전 서비스 공통 보안체계와 로그·감사 이력

근거: [RFP III-가-SER-CM-001~005, p.23~24]

  • 법령, 지침, 발주처 보안규정 준수
  • 개발보안, 인증, 접근통제, 암호화, 개인정보 보호, 관리자 권한 통제
  • 서버·네트워크 통신구간 암호화 및 키 관리
  • 사업자·투입인력 보안관리
  • 사용자 접속, 권한 변경, 데이터 처리, 배치·ETL, 오류, 주요 변경이력 로그관리

제안 배점 추정: 매우 높음

  • 보안 요구사항은 제안서 평가에서 별도 평가 관점으로 다뤄질 가능성이 높고, AI·데이터 플랫폼까지 포함된 사업 특성상 사고 영향도가 큼

구현 핵심 포인트:

  • Zero Trust 기반 접근통제: MFA, RBAC, PAM, 관리자 세션 기록
  • 전송구간 TLS 1.2 이상, 저장구간 KMS 암호화
  • SIEM 연계 가능한 구조화 로그(JSON) 표준
  • 개인정보/민감정보 데이터 흐름도와 보호대책 제출
  • SAST/DAST/SCA/Secret Scan을 CI/CD 품질 게이트에 내장

KEY-3. 테스트·성능·전환 통합 검증 체계

근거: [RFP III-가-TER-CM-001~007, p.21~22], [RFP III-가-PER-CM-001, p.20]

  • 단위테스트, 통합테스트, 성능테스트, 시스템테스트, 서비스 전환 계획 수립
  • 업무 시나리오 및 테스트 데이터 기반 수행
  • 요구사항별 적합 여부 확인
  • 사용자 화면 응답시간, 업무영역별 성능 적정성, 자원 사용현황 검증
  • 장애복구, 호환성, 웹접근성 등 시스템 운영 항목 검증

제안 배점 추정: 높음

  • 다수 서비스 병렬 구축에서 테스트 누락은 전환 실패로 직결된다. 요구사항 추적성 기반 테스트 매트릭스가 핵심 평가 포인트

구현 핵심 포인트:

  • RTM(Requirement Traceability Matrix) 기반 테스트 케이스 자동 매핑
  • 단위/통합/성능/보안/접근성 테스트 증적 저장소 운영
  • k6/JMeter 기반 부하 테스트, Lighthouse/axe 기반 웹 성능·접근성 점검
  • 전환 리허설 2회 이상, Rollback Runbook, 데이터 백업 체크리스트
  • P95 응답시간, 처리량, 오류율, CPU/Memory/DB Connection 기준 수립

KEY-4. 개발 표준, 전자정부 프레임워크, 모듈화 준수

근거: [RFP III-가-COR-CM-001~004, p.26~27]

  • 개발 표준 정의서 수립 및 준수
  • 정보시스템 구축 규정 및 기술표준 준수
  • 전자정부 표준 프레임워크 기반 개발
  • 공통코드, 공통기능, 공통모듈 재사용성과 일관성 고려

5. 차별화된 제안 전략 (Value Proposition)

V-1. Landing Zone Blueprint — "인프라 구축이 아니라 운영모델을 납품"

고객 Pain: 클라우드로 이전해도 계정·권한·네트워크·로그·비용 기준이 없으면 운영 복잡도만 증가한다.

제안 해법:

  • 착수 2주 내 Landing Zone Blueprint와 Account/Network/Security 표준안 확정
  • Terraform 기반 IaC로 네트워크, 보안그룹, 로깅, 모니터링, 백업 정책을 코드화
  • 서비스별 계정 분리와 중앙 보안/로그 계정 운영으로 감사 대응성 강화
  • 차별화 메시지: "서비스를 올릴 때마다 보안을 새로 설계하지 않는 플랫폼"

V-2. Quality Gate PMO — "검수 직전에 품질을 확인하지 않고 매주 품질을 통과"

고객 Pain: 대형 SI는 막판 통합테스트에서 결함이 폭증하고, 일정 지연과 범위 논쟁이 발생한다.

제안 해법:

  • 요구사항 ID별 RTM을 기준으로 설계·개발·테스트·결함·검수 증적 연결
  • CI/CD에 단위테스트, SAST, SCA, 컨테이너 취약점, OpenAPI 계약검증 자동화
  • 주간 품질 리포트: 요구사항 반영률, 테스트 통과율, 결함 Aging, 보안 취약점 잔여율
  • 차별화 메시지: "검수 가능한 증적을 개발 중에 계속 쌓는 방식"

V-3. Security-by-Default — "보안은 서비스별 선택사항이 아니라 공통 기본값"

고객 Pain: 서비스별 보안 구현 수준이 다르면 가장 약한 서비스가 전체 플랫폼의 사고 지점이 된다.

제안 해법:

  • 표준 인증/권한/감사로그 모듈을 공통 라이브러리로 제공
  • 관리자 행위 로그, 권한 변경 이력, 배치/ETL 이력, 데이터 처리 이력을 중앙 저장
  • 외부 접근, 관리자 접근, 시스템 간 접근을 분리하고 PAM/MFA/세션감사 적용
  • 차별화 메시지: "서비스가 늘어도 보안 수준은 흔들리지 않는다"

V-4. MCP-Ready Common Integration — "AI Agent 연계를 공통 인프라로 준비"

고객 Pain: Trade Agent 도입 후 각 서비스 API를 임의로 연결하면 도메인 확장마다 별도 개발과 보안 검토가 반복된다.

제안 해법:

  • 공통 API Gateway와 MCP 서버 표준을 함께 설계
  • 서비스 API를 MCP Tool Manifest로 등록하고 권한·Rate Limit·Audit을 공통 적용
  • 도메인 서비스별 MCP Adapter 템플릿 제공
  • 차별화 메시지: "AI Agent가 안전하게 호출할 수 있는 서비스 도구 생태계"

V-5. 2단계 RFP Readiness — "이번 구축 산출물이 다음 사업의 요구사항 자산"

고객 Pain: 1단계 사업이 끝나도 후속 사업 RFP 작성을 위해 현황·Gap·요구사항을 다시 수집해야 한다.

제안 해법:

  • 요구사항, 설계결정, 인터페이스, 운영지표, 미해결 과제를 구조화된 Backlog로 관리
  • 2단계 RFP 작성 지원을 위한 As-Is/To-Be/Gap/우선순위 템플릿 제공
  • 서비스별 고도화 후보와 예산 산정 근거를 운영 데이터 기반으로 제시
  • 차별화 메시지: "완료보고서가 아니라 다음 단계 착수 패키지를 남긴다"

7. 결론 및 제안 핵심 메시지

7.1 사업 본질 재정의

공통 영역은 "인프라와 관리 문서"가 아니라, 차세대 무역플랫폼 전체가 동일한 보안·품질·운영 기준으로 움직이게 만드는 Platform Foundation + Delivery Governance다. 공통 영역이 약하면 개별 서비스가 잘 만들어져도 전환, 운영, 보안, 감리 단계에서 전체 사업 리스크가 폭증한다.

7.2 제안 5축 메시지

  1. Landing Zone First — 멀티 Account, 네트워크, 보안, 로그, 비용관리 기준을 먼저 확정
  2. Security-by-Default — 인증·권한·암호화·감사로그를 공통 모듈과 운영정책으로 기본 적용
  3. Quality Gate PMO — 요구사항 추적 기반 테스트·보안·품질 증적을 매주 축적
  4. MCP-Ready Integration — Trade Agent 확장을 고려한 공통 API/MCP 연계 기반 선제 구축
  5. 2-Step Readiness — 1단계 산출물을 후속 RFP와 고도화 Backlog 자산으로 전환
