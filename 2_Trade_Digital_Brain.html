Trade Digital Brain 시스템 분석 보고서

1. 목표 시스템 정의

1-Liner Definition

Trade Digital Brain은 KITA 산하 8개 분산 서비스의 정형·비정형·반정형·행동 데이터를 단일 Data Lakehouse 위에 표준·메타·품질·리니지·라이프사이클 거버넌스 체계로 통합하고, AI Data Hub(Vector·Knowledge·Prompt 라이브러리)를 통해 Multi-Agent와 통합대시보드가 공통 활용 가능한 "AI Readable Data Foundation"을 구축하는 사업이다.

1.1 사업 배경 (Why)

  • As-Is 사일로 구조: KITA.net, tradeKorea, TradePro, 무역통계, 무역아카데미, 멤버십, KNEX(내부), 콜센터/SR이 별도 운영. 회원 식별·행동로그·콘텐츠 메타가 서비스 경계에 갇혀 있어 횡단 활용 불가 [RFP II-1, p.9]
  • 인프라 노후: 무역통계 운영계가 Solaris 11 / Oracle 19c / NetApp SAN 기반 온프레로 잔존, ETL은 InnoQuartz 7.2.1, 모니터링은 Zenius/Sherpa 등 레거시 솔루션 의존 [RFP II-2, p.10~14]
  • AI Data Readable 부재: AI 학습/추론에 직접 활용 가능한 표준화·벡터화 데이터 계층이 존재하지 않음. 정형/비정형/반정형 데이터의 파싱·청킹·임베딩 파이프라인이 미구축 [RFP III-나-DAR-DP-007, p.45~]
  • 거버넌스 공백: 데이터 표준(단어·용어·도메인·코드), 메타데이터 카탈로그, 데이터 리니지, 품질 검증, 라이프사이클 관리 체계가 전사 통합 수준으로 정립되지 않음 [RFP III-나-DAR-DP-001~017, p.42~50]

1.2 구축 목적 (What)

RFP I-3 사업목표 첫 번째 명문 [RFP I-3, p.1]:

"글로벌 무역 데이터 플랫폼 기반 무역 특화 지능형 Trade Agent 구축 — 통합데이터플랫폼 구축을 통해 무역통계 서비스를 고도화하고, 수출 기업의 전주기 업무 프로세스에 특화된 대화형 AI Trade Agent 구축"

RFP I-4 추진범위 첫 번째 항목 [RFP I-4, p.2~3]:

"Trade Digital Brain 구축 — 협회의 산재되어 있는 다양한 서비스의 분산된 데이터 이용 환경을 통합데이터 플랫폼으로 일원화하고, AI 서비스와 통합대시보드에서 데이터 활용이 가능한 환경을 구축"

1.3 최종 지향점 (Goal)

  • AI Readable Data Foundation: 모든 데이터가 표준·메타·벡터화 상태로 전환되어 27종 Trade Agent와 통합대시보드가 공통 소비 가능한 단일 데이터 기반 제공
  • 데이터 주권: 회원사 정보·매출·통상정보가 외부 LLM/CSP에 잔류하지 않는 폐쇄망 운영 가능 구조
  • 확장 가능 거버넌스: 신규 서비스(ESG·지재권 등) 추가 시 표준·메타·품질·리니지가 자동 적용되는 셀프 서비스 거버넌스
  • 레거시 무중단 전환: 현행 무역통계(Solaris/Oracle 19c) 서비스를 무중단으로 Lakehouse 환경으로 마이그레이션 [RFP III-나-PER-DP-002, p.40]

3. 핵심 요구사항 및 필수 고려사항 TOP 5

기술평가 100점 중 "기술 및 기능 25점", "성능 및 품질 15점", "프로젝트 지원 20점"의 핵심 가점은 L2 Lakehouse, L3 거버넌스, L1 ETL/문서처리, L4 데이터가상화, 무역통계 무중단 전환 5가지 영역에 집중된다.

KEY-1. Data Lakehouse 도입 + 분산 클러스터 아키텍처

근거: [RFP III-나-ECR-DP-001, p.32]

  • ACID 트랜잭션 지원, 메타데이터 기반 관리, 고속 분석 쿼리 명문 요구
  • 정형·비정형 분리 저장, 증분 적재, 이력 관리 필수
  • 클러스터 기반 분산 컴퓨팅, 부하 분산, 확장성 보장 명문

난이도/배점 추정: ★★★★★ (가장 핵심)

  • ECR 11개 중 1순위. 평가표 "기술 및 기능 25점"의 직접 대상
  • "구현 방안이 구체적으로 기술" 평가 기준에 정조준되는 영역. ADR(Architecture Decision Record) 형태로 채택안과 기각안의 근거를 명시 필요

구현 핵심 포인트:

  • Bronze(Raw) → Silver(Cleansed) → Gold(Curated) 3계층 메달리온 아키텍처
  • Apache Iceberg 또는 Delta Lake (ACID + Time Travel + Schema Evolution)
  • 컴퓨팅: Apache Spark + Trino(쿼리) 분리. 스토리지/컴퓨팅 디커플링
  • 오브젝트 스토리지: MinIO(S3 호환) 또는 KT Cloud Object Storage

KEY-2. 통합 메타데이터 관리 + 데이터 표준·품질·리니지·라이프사이클 거버넌스

근거: [RFP III-나-ECR-DP-003, p.36] + [DAR-DP-001 데이터 표준화, p.42] + [DAR-DP-007~008 데이터 품질·라이프사이클, p.45~46]

  • 메타데이터 자동 수집·등록·승인·감사 명문
  • 표준사전(단어·용어·도메인·코드) + DB Object 명명규칙 + 정합성 검증
  • 데이터 품질 검증 방안(검증 대상·시점·항목·방법) + 오류 식별·재검증 절차
  • 라이프사이클(생성·수집·적재·활용·보관·이관·삭제) 관리 + 개인정보 보관·삭제 기준
  • 메타데이터 기반 통합 거버넌스 + 데이터카탈로그·Biz메타 연계

난이도/배점 추정: ★★★★★

  • DAR(데이터 요구사항) 17개 중 다수가 거버넌스 영역. 평가표 "데이터 요구사항" 기준 평가 직접 매핑
  • 표준 정의 → 카탈로그 등록 → 품질 검증 → 변경 이력 → 라이프사이클이 순서적 의존 관계여서 일정 관리 난이도 매우 높음

구현 핵심 포인트:

  • 카탈로그: DataHub 또는 OpenMetadata (오픈소스 + 한국어 검증 사례 풍부)
  • 표준사전: DA# 또는 ERwin 기반 + 카탈로그 자동 동기화
  • 품질: Great Expectations + Soda Core, 임계치 위반 시 파이프라인 차단
  • 리니지: OpenLineage / Marquez (Spark·Airflow 자동 수집)
  • 라이프사이클: Iceberg Snapshot Expiration + Object Storage Lifecycle Policy

KEY-3. AI Data Hub — 비정형(특히 HWP/HWPX) 처리 + 임베딩 파이프라인

근거: [RFP III-다-SFR-AP-010 AI Data 파이프라인, p.55] + [SFR-AP-011 AI Data 전처리·가공, p.55~]

  • 정형·비정형·반정형을 표준 메타·AI 메타 표준에 따라 Parsing·Chunking·Embedding 명문
  • hwpx, hwp 포맷 지원 명문 (발주처 특성)
  • 복합 구성 문서(이미지·그래프·차트·표·문장)의 요소별 분리·구조 정보 보존
  • 데이터 분할 정보·원본 위치·식별 정보 메타화
  • 평가 데이터셋 구축 + 피드백 루프

난이도/배점 추정: ★★★★☆

  • HWP/HWPX 처리는 한국 공공기관 사업의 단골 위험 요소. 정확도 미달 시 Trade Agent 전체 응답 품질 하락
  • "발주처 특성을 고려"라는 명문이 들어 있어 평가위원의 특정 솔루션 사례 요구 가능성 큼

구현 핵심 포인트:

  • 문서 파서: Docling(IBM, 표·차트 인식 SOTA) + pyhwp/olefile(HWP/HWPX) + PyMuPDF(PDF) 다단 파이프라인
  • OCR: PaddleOCR 2.9 (한·영 동시 지원, GPU 가속)
  • Chunking: Recursive Character + Semantic Chunking 혼합. 표·차트는 별도 청크 분리
  • Embedding: BGE-M3(다국어) + KURE-v1(한국어 SOTA) 듀얼 모델
  • 파이프라인 오케스트레이션: Apache Airflow 2.10 DAG + Ray(분산 임베딩)

KEY-4. 데이터 가상화 + APIM (Single View 제공)

근거: [RFP III-나-ECR-DP-007 데이터가상화, p.40] + [ECR-DP-006 APIM, p.38]

  • 물리적 데이터 이동 없이 논리적 가상 뷰 생성·통합 명문
  • Row/Column 단위 보안 정책(RLS/CLS) 적용
  • JDBC/ODBC 표준 인터페이스 + Trade Agent·Dashboard 공통 활용
  • API 생명주기 관리, 트래픽 제어, 개발자 포털, 로그 감사

난이도/배점 추정: ★★★★☆

  • "AI Agent + Dashboard 공통 활용" 명문 → 단일 진실 소스(SSOT) 제공의 결정적 메커니즘
  • Trade Agent의 RAG·MCP 호출 정확도가 이 계층 품질에 직결

구현 핵심 포인트:

  • 데이터 가상화: Denodo 또는 Trino + Starburst(오픈소스 친화). 가상 뷰 메타는 카탈로그와 자동 동기화
  • APIM: Apache APISIX 3.x(오픈소스) 또는 Kong Gateway 3.8. 셀프 서비스 포털, 트래픽/할당량 관리, OAuth2/JWT 표준 인증
  • 보안 필터: SQL Injection 방어, 메시지 암복호화, IP 화이트리스트
  • 관찰성: OpenTelemetry Tracing → Grafana Tempo

KEY-5. 무역통계 시스템 무중단 마이그레이션 + 성능 SLA

근거: [RFP III-나-PER-DP-001/002, p.39~40]

  • 일반 조회 응답시간 3초, 복합 통계 5초 명문 (예시 기준, 발주처와 협의 확정)
  • 무역통계 마이그레이션 + 서비스 전환 + 롤백 절차 + 서비스 중단 허용 시간 명시 필요

난이도/배점 추정: ★★★★☆

  • 평가표 "성능 요구사항 15점" + "프로젝트 관리 15점" 동시 매핑
  • Solaris 11 / Oracle 19c / NetApp SAN의 운영계 데이터 → Lakehouse 환경으로 무중단 전환은 사업 후반부 최대 리스크 지점
  • ECR-DP-005 ETL 호환성 명문(기존 InnoQuartz 호환) 충족 필요 [RFP III-나-ECR-DP-005, p.37]

구현 핵심 포인트:

  • 전환 전략: Strangler Fig + Dual-Write (구·신 동시 적재 → 정합성 검증 → 컷오버)
  • 마이그레이션 도구: Oracle GoldenGate(CDC) 또는 Debezium for Oracle
  • 성능 검증: 일반 조회 P95 ≤ 3초, 복합 조회 P95 ≤ 5초, 동시 사용자 N명 → k6/JMeter 부하 테스트
  • 롤백: Lakehouse Time Travel + Iceberg Snapshot Rollback. 컷오버 후 N일 이내 원복 가능 보장

5. 차별화된 제안 전략 (Value Proposition)

V-1. Sovereign Data Foundation — "회원사 데이터가 미국 GPU에 한 줄도 남지 않는다"

고객 Pain: 회원사 매출·통상·개인정보의 외부 LLM 학습 데이터 잔류 리스크. RFP는 데이터 주권을 명문화하지 않았으나 발주처 담당자의 잠재 두려움 1순위.

제안 해법:

  • KT Cloud GPU + vLLM(self-hosted) + LiteLLM Proxy 3축으로 모든 추론을 국내 잔류
  • 외부 LLM 호출 시 Presidio PII 비식별 → LiteLLM Adapter → 응답 수신 후 재식별
  • 모든 트레이스를 Langfuse(self-hosted)에 기록, 외부 SaaS 추적 차단
  • 차별화 메시지: "발주처가 외부 LLM에 종속되지 않을 자유"를 데이터 플랫폼 본문에 박음

V-2. 기존 KITA 자산 무중단 통합 — "운영 단절 0의 전환"

고객 Pain: 기존 솔루션(InnoQuartz, Sherpa, Zenius, NetBackup, 휴네시온)에 운영 노하우가 축적되어 있으나 차세대 전환 시 단절되면 운영팀 적응 비용 폭증.

제안 해법:

  • ECR-DP-005 ETL 호환성 명문에 따라 InnoQuartz를 1차 ETL로 유지 + Airflow를 2차 오케스트레이터로 점진 도입(Strangler Fig)
  • DB 모니터링은 Sherpa/Zenius를 그대로 유지 + Prometheus/Grafana를 새 영역에만 도입
  • 백업은 NetBackup 10을 메인으로, Iceberg Snapshot은 보조로
  • 차별화 메시지: "혁신은 더하되 운영팀은 익숙한 도구를 잃지 않는다"

V-3. Eval-Driven Data Pipeline — "데이터 품질을 정량 SLA로 약속"

고객 Pain: 데이터 품질 검증을 RFP가 요구하지만, 대부분의 SI는 "QA 수행" 수준의 추상 답안만 제출. 발주처는 정량 지표를 보고 싶어 한다.

제안 해법:

  • Great Expectations 임계치(완전성·유일성·정합성·범위) 자동 검증 + 위반 시 파이프라인 차단(CD Gate)
  • Embedding Pipeline은 Ragas 0.2의 Context Precision/Recall 지표로 검증 → 임계치 미달 시 재처리
  • 분기별 발주처 합동 리뷰: 표준 적용률·품질 지표·리니지 커버리지 정량 SLA 보고
  • 차별화 메시지: "데이터 품질을 산문이 아닌 숫자로 보증한다"

V-4. AI-Ready Data Catalog — "Agent와 사람이 같은 카탈로그를 본다"

고객 Pain: 카탈로그는 통상 데이터 엔지니어용 도구로 머무름. 27 Trade Agent가 메타데이터를 직접 검색·활용하는 구조까지 가는 사례가 드물다.

제안 해법:

  • DataHub 메타데이터를 Vector화 → AI Data Hub에 저장 → Agent가 RAG로 메타 검색
  • "어떤 데이터가 어디 있는가"를 자연어로 질의 가능 (예: "2024년 베트남 수출 통계 테이블 어디 있어?")
  • 메타 변경 시 임베딩 자동 재생성 (CDC + Webhook)
  • 차별화 메시지: "AI가 직접 카탈로그를 읽는 데이터 플랫폼"

V-5. 단계형 4-Wave 이행안 — "12개월 빅뱅 금지, 안정적 단계 오픈"

고객 Pain: 12개월 내 8개 도메인 + 5계층 동시 신축 = 빅뱅 사고 두려움.

제안 해법:

  • Wave 1 (M1~M3): L1 인프라 + L2 Lakehouse MVP (5개 핵심 도메인 데이터 우선 적재)
  • Wave 2 (M3~M6): L3 거버넌스 + L4 APIM·데이터가상화 + 무역통계 마이그레이션 Dual-Write 시작
  • Wave 3 (M6~M9): AI Data Hub 정식 + Trade Agent 27종 RAG 연계 + 8개 도메인 MCP 노출
  • Wave 4 (M9~M12): 통합 테스트·전환 리허설·컷오버 + 안정화
  • 각 Wave 종료 시 발주처 Go/No-Go 게이트
  • 차별화 메시지: "사고 가능성을 12배 분산한다"

7. 결론 및 제안 핵심 메시지

7.1 사업 본질 재정의

Trade Digital Brain은 "데이터 플랫폼"이라는 일반 명사가 아니라, "27 Trade Agent + 통합대시보드 + 8개 도메인 서비스가 공통으로 소비 가능한 AI Readable Data Foundation"이다. 이 본질을 놓치면 단순 ETL/Lakehouse 사업으로 격하되어 평가에서 차별화 불가.

7.2 제안 3축 메시지

  1. Sovereign Data Foundation — KT Cloud GPU + vLLM + LiteLLM 3축으로 데이터 주권 보증
  2. Open + Compatible Stack — Apache Iceberg/Spark/Trino/DataHub 오픈소스 + InnoQuartz/Sherpa/Zenius/NetBackup 기존 자산 호환
  3. Eval-Driven Quality SLA — Great Expectations + Ragas로 데이터·임베딩 품질을 정량 SLA로 약속
