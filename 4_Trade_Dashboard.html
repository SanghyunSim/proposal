1. 목표 시스템 정의

1-Liner Definition

맞춤형 디지털무역 통합대시보드는 KITA 8개 분산 서비스의 콘텐츠·통계·진행상태·AI 결과를 회원사별 관심 컨텍스트(HS·국가·업종·세그먼트) 기반으로 단일 화면에 자동 구성하고, 콜드스타트 대응·자동 추천·맞춤형 알림·저장함 연계를 통해 수출기업 업무 전주기를 한 페이지에서 시작·재개할 수 있게 만드는 위젯 카탈로그 기반 초개인화 통합 포털이다.

1.1 사업 배경 (Why)

  • 메뉴 구조의 분산성: KITA.net 등 기존 서비스는 메뉴가 깊고 산재되어 있어 회원사가 "오늘 내가 봐야 할 정보가 무엇인지"를 한 화면에서 인지하지 못함. 정보 탐색 효율 저하 [ISP p.7, RFP II-1, p.9]
  • 개인화 부재: HS Code·관심국가·업종·이용이력 기반의 자동 위젯 구성·콘텐츠 추천 체계가 없어 모든 회원사가 동일 화면을 보는 상태. 신규/저활동 사용자에 대한 콜드스타트 전략 부재 [RFP III-라-SFR-DS-004, p.66]
  • 알림 채널 미통합: 규제 변경·정책·상담 진행 상태 등 시점성 정보를 SMS/메신저/App Push 다채널로 일관되게 전달할 수 있는 표준화된 알림 시스템 미구축 [RFP III-라-SFR-DS-006, p.67]
  • AI ↔ 화면 단절: Trade Agent가 생성한 추천·요약·브리핑이 대시보드 위젯에 직접 연계되지 않아, 사용자가 챗봇 화면을 별도로 탐색해야 활용 가능 [RFP III-라-INR-DS-004, p.69]

1.2 구축 목적 (What)

RFP I-4 추진범위 셋째 항목 [RFP I-4, p.3]:

"협회가 제공하는 다양한 무역 정보와 서비스를 사용자 관점으로 제공하고, 사용자별 관심사 맞춤형 정보를 통합하여 제공할 수 있는 환경을 대시보드로 구축. 사용자의 HS Code, 관심국가, 무역 주제 등 관심 정보를 회원정보·사용이력 등을 기반으로 자동 도출하여 개인화 맞춤 정보를 제공하는 통합대시보드 구축"

RFP III-라-INR-DS-001 [RFP III-라-INR-DS-001, p.69]:

"무역업무 관점의 핵심 정보/업무 상태를 한 화면에서 제공하는 통합 디지털 무역 대시보드 플랫폼 구축 방안을 제시"

1.3 최종 지향점 (Goal)

  • 위젯 카탈로그 기반 모듈형 화면: 약 20종 위젯(수출 유망 품목, 해외 바이어 매칭, 무역시장 브리핑, 사업/교육과정 추천 등)을 표준 규격(입력·출력·필터·권한·상태)으로 정의하고 운영 단계에서 자유롭게 추가·교체 가능 [RFP III-라-SFR-DS-001, p.65]
  • 세그먼트 + 행동 기반 초개인화: 회원 등급·산업·기업규모·HS·국가·이용이력을 종합한 사용자 세그먼트 자동 분류 + 콜드스타트 대응 + 점진적 개인화 전환 [RFP III-라-SFR-DS-004, p.66]
  • AI Agent ↔ Dashboard 단일 UX: Agent 결과(추천/요약/브리핑)를 위젯 인박스로 받고, 위젯에서 질문 입력→Agent 화면 전환→답변 저장함 복귀까지 단일 동선 [RFP III-라-INR-DS-004, p.69]
  • 이벤트 기반 멀티채널 알림: 변화·마감·상태 변경 트리거를 SMS/카톡 알림톡/App Push/이메일 등 다채널로 사용자 설정에 따라 차등 전달 [RFP III-라-SFR-DS-006, p.67]

3. 핵심 요구사항 및 필수 고려사항 TOP 5

평가표상 "기술 및 기능 25점" + "성능 및 품질 15점" + "프로젝트 지원 20점"의 직접 가산 영역은 위젯 표준 규격, 개인화·콜드스타트, 자동 추천+Agent 연계, 다채널 알림 트리거, 권한·접근통제 5축에 집중된다.

KEY-1. 위젯 카탈로그 + 표준 규격 + 메타 거버넌스

근거: [RFP III-라-SFR-DS-001, p.65] + [DAR-DS-001 위젯 데이터 정의, p.70]

  • 약 20종 위젯 카탈로그 개발 (수출 유망 품목·바이어 매칭·시장 브리핑·교육 추천 등)
  • 입력/출력/필터/권한/상태 처리 규격 표준화 → 일관된 개발 방식
  • 위젯 식별자·명칭·카테고리·연계 데이터 명세 등 최소 운영 메타 정의
  • 등록/수정/삭제·변경이력·버전 관리·배포 권한 관리 명문
  • 신규 위젯 추가는 SI 종료 후에도 카탈로그 메타 등록만으로 노출 가능 [RFP III-라-SFR-DS-008 운영관리, p.68]

난이도/배점 추정: ★★★★★

  • 평가표 "기능 요구사항·구체적 구현 방안" 직접 매핑. 위젯 표준 미정의 시 운영 단계에 진입할수록 화면 일관성·성능 모두 붕괴
  • 발주처 입장에서 "신규 위젯 추가 시 SI 재계약 없이 가능한가"가 ROI 핵심 질문

구현 핵심 포인트:

  • 위젯 표준 스펙: JSON Schema 기반 매니페스트 (입력 파라미터·출력 데이터 모델·필터·권한·상태·갱신주기)
  • Frontend 컴포넌트화: React Server Components + Suspense + Error Boundary 표준
  • 카탈로그 저장소: PostgreSQL + Object Storage(아이콘·미리보기)
  • 위젯 형상관리: Git 기반 + Semver(메이저·마이너·패치) + Canary 배포
  • Admin UI에서 등록/수정/사용중지/A/B 노출 비율 조정

KEY-2. 사용자 세그먼트 + 콜드스타트 대응 + 점진적 개인화

근거: [RFP III-라-SFR-DS-004 개인화 구성 및 콜드스타트, p.66]

  • 사용자 데이터 기반 관심 컨텍스트(HS/국가/업종) 자동 도출 + 사용자 편집 UX
  • 회원 등급·산업·기업규모 등 사용자 세그먼트 기준 정의 + 세그먼트별 맞춤 정보
  • 콜드스타트 대응: 신규/저활동 사용자 진입 시 기본 구성 + 관심정보 설정 유도 + 점진적 개인화 전환 명문

난이도/배점 추정: ★★★★★

  • 가장 차별화 가능한 영역. SI사 대부분이 정성적 답안만 제출함
  • 콜드스타트 전략(Bootstrap → Implicit → Explicit) 3단 진화는 구체성으로 평가 가산점

구현 핵심 포인트:

  • 세그먼트 정의: 룰 기반(회원 등급·산업·기업규모) + 클러스터링(K-means / DBSCAN) 혼합
  • 콜드스타트 3단 전략:
  1. Bootstrap (가입~30일): 세그먼트 기본 위젯 패키지 + 관심 설정 UX 유도
  2. Implicit (행동 누적): 클릭/드릴/체류시간 기반 위젯 가중치 동적 조정
  3. Explicit (수동 편집): 사용자 직접 위젯 추가/제거 (SFR-DS-007 My 대시보드)
  • Feature Store: Feast 0.40 + Redis(Online Store) + ClickHouse(Offline)
  • 동시 운영: 세그먼트 단위 A/B 테스트(GrowthBook) → 효과 측정 후 정책 반영

KEY-3. 자동 추천 + 신뢰도·후행 동선 + AI Agent 연계

근거: [RFP III-라-SFR-DS-005 콘텐츠/사업 자동 추천, p.66~67] + [INR-DS-004 AI Agent 연계, p.69]

  • 개인화 기반 관심 콘텐츠(사업/교육/상담/바우처/뉴스) 자동 추천
  • 통계/분석 결과 기반 수출입 유망시장/품목 추천
  • 추천 결과 신뢰도 확보 방안 + 후행 버튼(신청/상담) 명문
  • Agent 생성 요약/브리핑/추천을 위젯에서 조회/표시 + Deeplink 화면 전환 + 저장함 연계

난이도/배점 추정: ★★★★☆

  • 추천 신뢰도와 후행 전환율은 발주처가 운영 후 가장 먼저 측정하는 KPI
  • INR-DS-004 명문이 Agent와 Dashboard의 단절 없는 UX 요구 → 시연 시 임팩트 큰 영역

구현 핵심 포인트:

  • 추천 모델 3축 혼합:
  1. 콘텐츠 기반: TF-IDF + Vector(BGE-M3) 코사인 유사도
  2. 협업 필터링: LightGBM/XGBoost (회원 행동 매트릭스)
  3. 룰 기반: 운영 정책 강제 노출(공지·캠페인 우선)
  • 신뢰도 표기: 모델 점수 → 백분율 + "왜 추천되었나" Explain 텍스트(SHAP·룰 명시)
  • Agent 연계: 위젯 우측 상단 "AI에게 묻기" 버튼 → Deeplink로 Trade Agent에 사용자 컨텍스트(기업·관심분야) 전달 → 응답 후 "저장함에 저장" → 대시보드 인박스에 자동 추가
  • 후행 버튼: 위젯 카드 하단 CTA 표준화 (신청·상담·바로가기)

KEY-4. 맞춤형 알림 (이벤트 트리거 + 다채널 정책)

근거: [RFP III-라-SFR-DS-006 맞춤형 알림 서비스, p.67] + [SFR-DS-007 사용자 설정/관리, p.67]

  • 변화·마감·상태 변경 등 이벤트 트리거 기준 정의 명문
  • 알림 대상/채널/빈도 사용자 맞춤 설정
  • 알림 히스토리 조회·관리

난이도/배점 추정: ★★★☆☆

  • 평가표 "기능 요구사항"의 일부지만, 다채널 발송·이벤트 트리거·구독 관리는 SI 일반 기능 수준
  • 차별화는 "트리거 정의의 구체성"과 "전송량 제어"에 있음

구현 핵심 포인트:

  • 이벤트 트리거 카탈로그: 변화(데이터 임계치), 마감(D-N), 상태 변경(반려·승인·발급) 등 표준 이벤트 타입 약 30~50종 정의
  • 메시지 큐: Apache Kafka 3.7 Event Bus + Consumer Group 별 채널 어댑터
  • 채널 어댑터: FCM(Android) / APNs(iOS) / 카카오 알림톡 / SMS / Email
  • Quiet Hours·Frequency Cap: 사용자별 빈도 제한 (Rate Limiter + Token Bucket)
  • 발송 추적: Delivery Receipt 수집 → 미수신 시 Fallback 채널(SMS) 자동 전환
  • 메시지 템플릿: 변수 치환 + 다국어(한·영) + A/B 테스트

KEY-5. 권한 + 데이터 접근 통제(RLS) + 행동 로그 + 모니터링

근거: [RFP III-라-SER-DS-001, p.71] + [QUR-DS-001 서비스 모니터링 및 사용 로그 수집, p.72]

  • 사용자/관리자 역할 구분 + 권한 관리
  • 기업/사용자별 접근통제(RLS) 준수, 조회 요청 시 기업/사용자 ID 컨텍스트 적용
  • 다운로드 권한 기반 허용/차단 정책
  • 권한 변경·다운로드·관리자 행위 감사 로그
  • 사용률·전환·구독·오류·지연 위젯 등 운영 지표 모니터링
  • 위젯 노출/클릭/필터/드릴/알림 행동 이벤트 정의 + 수집 + 사용지표 리포트

난이도/배점 추정: ★★★★☆

  • 보안 SER 2개 + 품질 QUR 1개 직접 매핑
  • 운영 KPI 측정 인프라 미구축 시 V-3(Eval-Driven SLA) 메시지 무력화

구현 핵심 포인트:

  • 권한: Keycloak 26 + JWT 클레임에 사용자ID·기업ID·역할 포함

5. 차별화된 제안 전략 (Value Proposition)

V-1. Catalog-Driven Dashboard — "운영자가 SI 없이 위젯을 관리한다"

고객 Pain: 신규 위젯 추가·기존 위젯 수정 시 SI사에 재계약·재개발 의존. 운영 자율성 부재.

제안 해법:

  • 위젯 표준 규격(JSON Schema 매니페스트) + Admin Console + 카탈로그 버전 관리(Semver) + Canary 배포
  • 운영자가 카탈로그에서 직접 위젯 등록/노출/A/B/중지 가능
  • "신규 위젯 1개 추가에 필요한 운영자 작업 시간"을 15분 이하로 SLA 약속
  • 차별화 메시지: "발주처가 운영 단계에서 위젯을 직접 키울 수 있다"

V-2. Cold-to-Warm Personalization — "가입 1일차부터 의미 있는 화면"

고객 Pain: 신규/저활동 사용자가 빈 대시보드를 마주하면 즉시 이탈. SI 결과물의 가장 큰 사용성 함정.

제안 해법:

  • Bootstrap → Implicit → Explicit 3단 콜드스타트 전략
  • 가입 페이지에서 1분 내 4~5개 핵심 관심사 입력 → 세그먼트 자동 분류 → 즉시 개인화 위젯 8~10개 노출
  • 가입 30일 이내 행동 데이터로 점진적 가중치 조정
  • 운영팀이 세그먼트별 위젯 디폴트를 GrowthBook으로 직접 실험·반영
  • 차별화 메시지: "신규 회원이 첫날부터 빈 화면이 아닌 자기 화면을 본다"

V-3. Dashboard-as-Agent-Inbox — "AI가 만든 답을 화면이 받는다"

고객 Pain: AI Agent와 대시보드가 분리되어 있으면 사용자가 두 화면을 번갈아 탐색해야 함. RFP가 INR-DS-004로 명문 요구하지만 SI 대부분은 단순 Deeplink 수준만 구현.

제안 해법:

  • "AI 무역 비서 인박스" 위젯 표준 제공 — Trade Agent의 추천·요약·브리핑이 자동으로 인박스 카드로 도착
  • 위젯 카드에 "전체 보기"(Agent 화면 전환), "저장", "후속 질문" CTA
  • 사용자가 Agent 화면에서 받은 답변을 "저장함"에 넣으면 다음 로그인 시 대시보드 인박스에 자동 노출
  • 차별화 메시지: "Agent 화면을 열지 않아도 AI 결과가 매일 나에게 온다"

V-4. Trigger-Catalog Notification — "알림이 시끄럽지 않은 시스템"

고객 Pain: 다채널 알림 시스템은 운영 후 N개월 내 "알림 피로" 문제로 사용자 차단·구독 해제 폭증.

제안 해법:

  • 30~50종 표준 트리거 카탈로그 + 사용자별 Quiet Hours·Frequency Cap·우선순위 관리
  • 카카오 알림톡(필수 알림) → App Push(중요 알림) → SMS(폴백) → Email(보고서) 채널 우선순위 표준화
  • 발송 후 24h 내 클릭 없으면 Frequency 자동 감소(머신 러닝 적응형)
  • 분기별 알림 전환율 합동 리뷰 + 트리거 정책 정제
  • 차별화 메시지: "알림이 늘어도 사용자가 떠나지 않는다"

V-5. Eval-Driven UX SLA — "사용성 지표를 정량 SLA로 약속"

고객 Pain: 대시보드 사용성은 정성적 만족도로만 측정되어 운영 단계에서 개선 근거가 없음.

제안 해법:

  • 핵심 지표 SLA 명시: 첫 화면 로딩 P95 ≤ 2초 / 위젯별 응답 P95 ≤ 1.5초 / 콜드스타트 30일 내 첫 위젯 클릭률 ≥ 60% / 추천 후행 전환율 ≥ 8%
  • ClickHouse + Grafana로 위젯별 사용지표 실시간 시각화
  • 분기별 발주처 합동 리뷰 + 하위 20% 위젯 자동 재설계 큐 등록
  • 차별화 메시지: "대시보드 운영을 데이터로 한다"

R10운영자 콘솔 사용성 부족 — Admin이 카탈로그·트리거·세그먼트를 직접 운영 못 하면 V-1 무력화MedMedMED운영자 페르소나 정의 + 핵심 동선 5종 워크플로우 사전 정의. 사업 종료 전 운영자 핸즈온 교육 4회
R11세그먼트 정의 합의 지연 — KITA 부서 간 세그먼트 기준 합의 실패 시 콜드스타트 무력MedMedMED사업 초기 데이터 거버넌스 협의체 + ISP 산출물 활용. 룰 기반 세그먼트(회원 등급·산업) 우선 적용 후 클러스터링 점진 도입
R12알림 채널 사업자 종속 — 카카오 알림톡 정책 변경 시 핵심 알림 마비LowMedLOW채널 추상화 레이어(Notification Hub) + Fallback 채널 자동 전환. 카카오 알림톡 → SMS → Email 우선순위

7. 결론 및 제안 핵심 메시지

7.1 사업 본질 재정의

통합대시보드는 "메인 화면 개편"이나 "포털 통합"이 아니라, "위젯 카탈로그 + 세그먼트 개인화 + AI Agent 인박스 + 다채널 알림"이 한 시스템으로 통합된 모듈형 초개인화 포털이다. 단순 화면 개편으로 접근하면 평가 단계에서 차별화 불가.

7.2 제안 5축 메시지

  1. Catalog-Driven — 운영자가 SI 없이 위젯을 직접 관리
  2. Cold-to-Warm — 가입 1일차부터 의미 있는 화면
  3. Dashboard-as-Agent-Inbox — AI 결과가 매일 도착하는 인박스
  4. Trigger-Catalog Notification — 알림 피로 없는 다채널 시스템
  5. Eval-Driven UX SLA — 사용성 지표를 정량 SLA로 약속
