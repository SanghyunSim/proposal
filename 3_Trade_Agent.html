Trade Agent 시스템 분석 보고서

1. 목표 시스템 정의

1-Liner Definition

Trade Agent는 KITA 회원기업의 무역 업무 전주기를 지원하기 위해 Master/Orchestration + 27종 Unit Agent(범용 16 + 도메인 11)를 LLM 추상화·MCP·RAG·가드레일·평가체계 위에 통합한 Multi-Agent 기반 무역 특화 지능형 비서 시스템이다.

1.1 사업 배경 (Why)

  • 단발성·고립형 AI: 현재 KITA AI 서비스는 TradePro 1:1 상담 영역의 단순 챗봇 수준에 한정. HS Code 매칭, 바이어 추천, 서류 자동 심사, ESG 진단, 지재권 모니터링 등 무역 도메인 핵심 시나리오는 미구현 [RFP II-1, p.9]
  • 데이터 단절: 분산된 8개 서비스의 데이터·메타·지식이 통합되어 있지 않아 AI가 "어디에 무슨 데이터가 있는지"를 인지하지 못함. RAG·Tool Use를 위한 표준 데이터 계층이 부재 [RFP III-다-SFR-AP-001, p.51]
  • 모델 종속 리스크: 특정 외부 LLM에 종속될 경우 정책 변경·가격 인상·데이터 주권 이슈에 취약. RFP가 명시적으로 "특정 모델 종속형 아키텍처 지양"을 요구 [RFP III-다-SFR-AP-001, p.52]
  • 보안·윤리 공백: 가드레일·프롬프트 인젝션 방어·PII 비식별·감사 로그 등 LLM 운영의 표준 보안 체계가 미구축 → 회원사 데이터 외부 유출 시 신뢰 사고 직결 [RFP III-다-SER-AP-001~004, p.62~64]

1.2 구축 목적 (What)

RFP I-3 사업목표 첫 번째 항목 [RFP I-3, p.1]:

"수출 기업의 전주기 업무 프로세스에 특화된 대화형 AI Trade Agent 구축 — 분산된 데이터의 통합 View 제공 및 AI Agent 연계를 통해 수출 기업의 관심·행동 기반 맞춤형 대시보드 및 알림 서비스 제공"

RFP I-4 추진범위 둘째 항목 [RFP I-4, p.3]:

"Multi Agent 기반의 통합 Agent 서비스를 구축하여 협회의 다양한 서비스를 원페이지 통합뷰 기반의 대화형 환경에서 정보를 제공하고 사용자의 업무를 지원"

RFP III-다-SFR-AP-001 [RFP III-다-SFR-AP-001, p.51~52]:

"범용 기능 Agent 16종 개발 및 도메인 특화 Agent 11종 개발. LLM·데이터·템플릿·규칙 기반 기능 등 전 유형의 Unit Agent에 공통으로 적용되는 표준 아키텍처가 제시되어야 함"

1.3 최종 지향점 (Goal)

  • Domain-Native AI Workforce: 무역 도메인 27 Unit Agent가 협력하여 수출기업 업무 시나리오(HS 매칭 → 관세규제 → 바이어 매칭 → 수출실적 증명 → ESG 진단)를 단일 대화로 끝낼 수 있는 통합 환경
  • Vendor-Neutral LLM Layer: LiteLLM 기반 모델 추상화로 self-hosted(vLLM) ↔ HyperCLOVA ↔ GPT/Claude 무중단 교체 가능
  • Sovereign & Trustworthy AI: 가드레일·PII 비식별·프롬프트 인젝션 방어·감사 로그를 표준화하여 데이터 주권과 신뢰성 동시 보증
  • Eval-Driven Operation: Ragas·DeepEval 기반 정량 평가 자동화로 운영 중 지속 품질 개선 가능 (HITL 피드백 루프)

3. 핵심 요구사항 및 필수 고려사항 TOP 5

기술평가 100점 중 "기술 및 기능 25점"의 직접 대상이며, 27 Unit Agent의 표준 아키텍처·Orchestration·서비스 특화·보안·성능 5축이 평가 가중치의 70% 이상을 가져간다.

KEY-1. Agent 표준 아키텍처 + LLM 추상화 (모델 종속 회피)

근거: [RFP III-다-SFR-AP-001, p.51~52]

  • 27 Unit Agent 전체에 공통 적용되는 표준 아키텍처(시스템 프롬프트·지식데이터 계층·파이프라인) 명문 요구
  • "각 Agent들이 정의된 역할과 수행 작업의 성격에 맞게 LLM모델을 선택하고 교체할 수 있게 구현되어야 함 (특정 모델 종속형 아키텍처 지양)" 명문
  • "모델 추상화를 통해 추가 LLM 모델 도입 시 추가 개발 없이 연동 가능한 구조" 명문
  • Multi Modal 지원(문서·이미지·음성), 토큰 최적화(캐싱·로그·Context 관리), 평가 지표 정의

난이도/배점 추정: ★★★★★

  • SFR-AP-001은 27 Agent 전체의 공통 설계서. 평가위원이 "표준 아키텍처 다이어그램"과 "모델 교체 데모"를 직접 요구할 가능성 매우 높음
  • 추상화 레이어 미흡 시 향후 LLM 가격 변동·정책 변경에 대응 불가 → 발주처 장기 ROI 직격

구현 핵심 포인트:

  • LLM Gateway: LiteLLM Proxy (OpenAI 호환 API, 라우팅·폴백·Spend Tracker)
  • 표준 Agent 골격: 시스템 프롬프트 분리(YAML) + 지식 어셈블리(LangChain Retriever) + Tool 매니페스트(JSON Schema) + 평가 어댑터
  • 토큰 최적화: GPTCache + Semantic Cache (의미 기반 캐시) → 30~50% 비용 절감
  • 모델 풀: vLLM(self-hosted Llama 3.3 70B / Qwen 2.5 / EXAONE 3.5) + 외부(HyperCLOVA·GPT-4o·Claude·Gemini)

KEY-2. Multi-Agent Orchestration (A2A · 워크플로우 · 컨텍스트)

근거: [RFP III-다-SFR-AP-002, p.52~53] + [SFR-AP-008 Master Agent, p.54]

  • 다양한 A2A 아키텍처 검토 후 발주처 환경에 적합한 Orchestration 구조 제시 명문
  • 작업 순서·의존성·분기·오류 대응·Fallback 시나리오 정의
  • 작업 진행 상태와 맥락의 멀티턴 유지 + 세션 초기화 후에도 과거 대화 호출 시 맥락 복원
  • 중앙 집중형(Orchestration) vs 분산형(Choreography) 적합 구조 선택 + 근거 명시
  • Orchestration 레벨에서 전체 작업 이력 관리 + Agent별 행동 모니터링 + 평가

난이도/배점 추정: ★★★★★

  • SI 시장에서 가장 많이 실패하는 영역. 단일 Agent → Multi-Agent 확장 시 컨텍스트 폭주, 무한 호출, 상태 동기화 실패가 빈번
  • 중앙형 vs 분산형 선택의 명확한 ADR(Architecture Decision Record)이 평가위원 질문 방어용 필수

구현 핵심 포인트:

  • 메인 Orchestrator: LangGraph 0.2 StateGraph (체크포인트·재시작·시각화 지원, Trace 표준 호환)
  • 보조 패턴: CrewAI (Role-based 협업, 도메인 특화 Agent 그룹)
  • State Store: PostgreSQL Checkpointer + mem0 (장기 기억) + Redis(단기 세션)
  • Fallback 전략: Tool 호출 실패 → Retry(지수 백오프) → 대체 Agent → 사용자 가이드 메시지
  • 시나리오 데이터화: 워크플로우를 JSON으로 정의해 신규 시나리오 추가 시 코드 수정 없이 적용 (DAR-AP-002 명문)

KEY-3. 서비스 특화 Agent — 수출실적증명·ESG·지재권 (도메인 깊이)

근거: [RFP III-다-SFR-AP-009, p.57~58]

  • 수출실적증명 자동화: HWP/HWPX/PDF 등 다양한 양식 자동 인식 + 필드 입력 검증 + 심사 규칙 기반 판정 + 우선순위 추천 + 담당자 자동 통보
  • ESG 전문 Agent: VSME 기반 진단, 약식 ESG 보고서(XBRL) 작성 지원, 1:1 상담 연계
  • 지재권 전문 Agent: 지재권 특화 상담, 서류 초안 작성, IP 사업화 지원, 법률 상담

난이도/배점 추정: ★★★★★

  • 도메인 깊이가 SI사 공통 약점. 평가위원의 "정말 무역을 아는가"라는 질문의 직접 대상
  • HWP/HWPX 자동 인식 정확도가 부족하면 Trade Agent 전체 신뢰도 동시 하락 (도미노 리스크)
  • ESG VSME / XBRL은 한국 SI 사례가 매우 적은 신영역

구현 핵심 포인트:

  • 문서 인식: Docling(IBM, 표·차트·레이아웃 SOTA) + pyhwp / olefile(HWP·HWPX) + PyMuPDF(PDF) + PaddleOCR 2.9(한·영) 다단 파이프라인
  • 검증 규칙: 룰 엔진(Drools or 자체) + LLM as Judge 이중 검증
  • HITL 루프: 자동심사 후 담당자 확인 단계 필수 (RFP "사용자 확인 및 심사 결과 발급" 명문)
  • ESG: VSME 질문 트리(인터랙티브) + Argilla 라벨링 → XBRL 보고서 자동 생성

5. 차별화된 제안 전략 (Value Proposition)

V-1. Vendor-Neutral LLM Stack — "모델 자유도가 곧 발주처의 협상력"

고객 Pain: 특정 LLM 종속 시 가격 인상·정책 변경·EOS에 발주처가 무력. RFP가 명문으로 종속 회피를 요구.

제안 해법:

  • LiteLLM Proxy 단일 게이트웨이로 self-hosted(vLLM) ↔ HyperCLOVA ↔ GPT-4o ↔ Claude ↔ Gemini 동일 인터페이스
  • 발표 시 GPT-4o → EXAONE 3.5 실시간 무중단 스왑 데모 (코드 수정 없이 환경변수만 변경)
  • LiteLLM Spend Tracker로 모델별 일일 비용 추이 시각화 → 발주처가 직접 비용 통제
  • 차별화 메시지: "발주처가 1년 후 어떤 LLM을 쓸지 미리 결정하지 않아도 된다"

V-2. Sovereign Trade Agent — "회원사 데이터가 미국·유럽 GPU에 한 줄도 남지 않는다"

고객 Pain: 회원사 매출·통상·개인정보의 외부 LLM 학습 데이터 잔류 + 외부 SaaS Trace 시스템 송출 리스크.

제안 해법:

  • KT Cloud GPU + vLLM(self-hosted) + Langfuse(self-hosted) 3축으로 모든 추론·트레이스를 국내 잔류
  • 외부 LLM 호출 필요 시 Presidio + Rebuff 2단 비식별 → LiteLLM Adapter → 응답 후 재식별 매핑
  • SER-AP-002 "학습 데이터 유출 방지" 명문 충족: 외부 LLM 호출 시 enable_training=false 강제 + 호출 페이로드 감사 로그
  • 차별화 메시지: 메가존(AWS Bedrock 의존)과 정면 차별화

V-3. Eval-Driven Agent Quality SLA — "Agent 품질을 매주 정량 지표로 약속"

고객 Pain: Agent 품질이 정성적 표현("정확하게", "빠르게")으로만 약속되는 SI 관행. RFP가 "모호 표현 불가능 간주" 룰로 정조준.

제안 해법:

  • 모든 Agent에 Ragas + DeepEval 정량 임계치 자동 검증: Faithfulness ≥ 0.85, Context Precision ≥ 0.80, Answer Relevancy ≥ 0.85, TTFT ≤ 1.5s
  • 임계치 미달 시 CD Gate 차단(배포 불가) + Slack/이메일 알림
  • 분기별 발주처 합동 리뷰: 27 Agent별 점수 추이 + 개선 액션 + 평가 데이터셋 인계
  • 차별화 메시지: "Agent 품질은 SLA 숫자로 약속한다"

V-4. MCP-First Architecture — "신규 도메인은 코드 수정 없이 추가"

고객 Pain: 신규 서비스 추가 시 Agent 코드 수정·재배포 필요한 구조 → 운영 부담 폭증.

제안 해법:

  • Anthropic MCP(Model Context Protocol) 표준 채택 → 8개 KITA 도메인을 MCP 서버로 노출
  • 신규 도메인 추가 시 MCP 서버만 구현하면 모든 Agent가 즉시 활용 (Tool Registry 자동 등록)
  • 외부 시스템(관세청·FTA·KTNET 등)도 MCP Adapter 패턴으로 표준화
  • 차별화 메시지: "운영 단계에서 도메인 확장 시 SI 재계약 없이 자체 추가 가능"

V-5. Trade-Native Agent Library — "27 Agent 중 11개를 사전 검증한 골든 데이터셋과 함께 제공"

고객 Pain: SI 종료 후 Agent 품질 검증·개선이 발주처 부담으로 전가됨.

제안 해법:

  • 27 Agent별 골든 데이터셋(각 50~200건) 사업 수행 중 구축 → 발주처 인계
  • 도메인 11 Agent(수출실적증명·ESG·지재권 등)는 사전 PoC 검증 데이터셋과 함께 제공
  • Argilla 기반 HITL 라벨링 인프라 + 운영팀 교육 자료 패키지
  • 차별화 메시지: "Agent 운영 자산을 발주처가 직접 키울 수 있는 도구까지 인계"

7. 결론 및 제안 핵심 메시지

7.1 사업 본질 재정의

Trade Agent는 "LLM 챗봇 도입"이 아니라, "27 Unit Agent + Master + Orchestration + LLM 추상화 + RAG + MCP + 가드레일 + 평가체계"가 한 시스템으로 통합된 Multi-Agent SI다. 단일 LLM 통합으로 접근하는 SI는 평가 단계에서 탈락한다.

7.2 제안 5축 메시지

  1. Vendor-Neutral LLM Stack — LiteLLM 추상화로 모델 자유도 = 발주처 장기 협상력
  2. Sovereign Agent — 회원사 데이터가 외부 LLM·SaaS Trace에 잔류하지 않는 구조
  3. Eval-Driven Quality SLA — 27 Agent별 정량 임계치를 분기 SLA로 약속
  4. MCP-First Architecture — 신규 도메인을 코드 수정 없이 추가 가능
  5. Trade-Native Agent Library — 골든 데이터셋·HITL 인프라까지 운영 자산으로 인계
